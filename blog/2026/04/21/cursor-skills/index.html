Cursor Skill 怎么写、放哪、怎么用

发表于2026-04-21|更新于2026-05-11|AI
|阅读量:

在 Cursor 里，Skill 是一组给 Agent 看的说明文件（常见为 SKILL.md），用来在特定任务场景下 注入领域知识、流程约束、项目惯例。它不是「魔法提示词」，而是 可版本化、可复用的协作规范：你写清楚，模型就更不容易跑偏。

下面分 定位、和 Rules 的分工、文档结构、一个可抄的骨架、维护策略 几部分写深一点。

一、Skill 解决什么问题

  • 团队里重复的「每次都要交代一遍」：分支策略、提交信息格式、目录约定、发布节奏。
  • 某类任务的固定步骤：发版检查清单、接口联调顺序、写测试的最低要求。
  • 工具链说明：怎么调内部脚本、环境变量名、常见坑、回滚方式。
  • 跨仓库可复用的方法论：例如「如何 triage PR、如何读 CI 日志」——与具体业务代码解耦时特别适合 Skill。

不适合塞进 Skill 的：动辄上千行的业务背景、会频繁过期的临时排期——这些应放在 Wiki/工单，Skill 里只留 链接与触发条件。

二、和 Project Rules、User Rules 怎么分工（经验法则）

载体 更适合放什么 特点
User Rules 个人习惯、全局语气、通用禁忌 跟人走，不一定进 Git
Project Rules（.cursor/rules 等） 本仓库 的架构边界、技术栈、目录含义 应进版本库，Code Review 可审
Skill（SKILL.md） 某一类任务 的 SOP、检查表、与外部系统的交互说明 可拆成多份，按需引用

粗记：Rules 定义「在这个仓库里一般怎么做事」；Skill 定义「做这类任务时额外打开哪本手册」。

三、一般放在哪里

常见做法包括（以你当前 Cursor 版本界面为准，路径可能微调）：

  • 用户目录下由 Cursor 管理的 skills 目录（多 Skill 分子文件夹，每份一个 SKILL.md）；
  • 与 Project Rules 互相引用：在规则里写「当任务涉及 X 时先读某 Skill」。

关键：Agent 能稳定 发现与读取 该 Markdown；若路径生僻，在 Rules 里写死「绝对或相对路径 + 何时读」。

四、SKILL.md 建议结构（可执行优先）

  1. 何时启用：一句话，触发条件写清楚（关键词、目录、任务类型）。
  2. 目标与禁止项：要达成什么；禁止改哪些目录、哪些依赖、哪些公开 API。
  3. 步骤清单：编号、可勾选；每步对应 可验证 的产物（命令、文件、截图）。
  4. 与仓库的对接：路径、脚本名、package.json script、环境变量。
  5. 验收标准：测试命令、lint、是否需要更新 changelog。
  6. 故障与回滚：常见错误码、找谁、如何撤销。

篇幅宜 短而可执行；太长时 拆 Skill，用主 Skill 做索引。

六、使用方式（对使用者）

  • 对话里 点名 Skill（「按某某 Skill 执行」）；
  • 或在 Project Rules 里默认关联某类路径（例如 **/migrations/** 触发数据库相关 Skill）；
  • 任务或流程变更时 同步改 Skill，并在 PR 里 可见地 diff——避免文档与代码分叉。

七、维护策略

  • 季度Review：删掉不再执行的步骤；合并重复的 Skill。
  • 版本化：重大流程变更时可在 Skill 标题注明 v2 或生效日期。
  • 可测试：若 Skill 里写了命令，至少在文档里注明 在 CI 里是否已有等价步骤。
