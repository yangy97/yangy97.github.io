企业级事故响应与 On-Call：分级、指挥官与无责复盘

发表于2026-04-23|工程化
|阅读量:

线上事故在企业里不仅是「技术问题」，还涉及 客户沟通、合规通报、值班疲劳。可观测性解决「看见」；事故响应解决 谁在什么时间点做什么、如何收敛、如何防止再犯。下面写 严重级别、角色分工、Runbook、复盘模板，并给 虚构但可套用的例子。

一、严重级别（示例分级，需与业务对齐）

级别 典型场景 响应期望（示例）
SEV1 核心交易全挂、大面积数据损坏风险 15 分钟内有人接手，成立应急组
SEV2 重要功能降级、部分用户不可用 30 分钟内响应，值班 + 相关域负责人
SEV3 边缘功能、可 workaround 工单排期，工作时间内处理

要点：级别定义要和 产品/客服 一致，避免「研发觉得是 SEV2，对外已群情激愤」。

二、角色：指挥官（Incident Commander）不是「最资深开发」

指挥官负责：定优先级、协调人、对外口径、何时升级/何时结束，不必亲自改代码。
技术负责人（或轮值）专注 根因与缓解。
沟通负责人（可选）更新 状态页 / 客户群。

反模式：所有人都在查日志，没人 和客户经理同步预计恢复时间 → 信任危机 不亚于技术本身。

三、Runbook：告警链接到「第一步做什么」

每条 P1/P2 告警 建议关联 简短 Runbook：

  1. 确认影响：哪些区域、哪些客户、是否 数据面 问题。
  2. 止血：扩容、回滚、关特性开关、切只读（按预案）。
  3. 排查：入口 5xx → 下游依赖 → 最近发布 / 配置变更（与 变更管理系统 对齐）。
  4. 升级：何时拉 DBA / 网络 / 云厂商 TAM。

示例（片段）：「payment-service 5xx > 5%」

  • 看网关 → payment 还是 下游 PSP；
  • 若 PSP 状态页正常，查 最近 deploy / 密钥轮换；
  • 若确认是 坏发布，执行 回滚 Runbook（与 CI/CD、发布策略篇一致）。

四、示例：一次「数据库连接池打满」的时间线（虚构）

时间 事件
10:00 订单接口 P99 飙升，告警触发
10:08 确认 应用侧 连接池等待队列长，DB CPU 正常
10:15 发现新版本 某循环未关闭游标（代码缺陷）
10:20 回滚 上一版本镜像；连接释放，指标恢复
同日 事后复盘：为何预发未发现？→ staging 流量模型 不足，补充 压测场景

五、无责复盘（Blameless Postmortem）写什么

推荐结构：

  1. 摘要：影响时长、影响用户范围、是否违反 SLO。
  2. 时间线：发现 → 缓解 → 完全恢复（精确到分钟更好）。
  3. 根因：用 五个为什么 或 鱼骨图，区分 触发条件 与 系统性原因（如缺少测试类型）。
  4. 行动项：每项 负责人 + 截止日期；区分 立即 / 30 天内 / 季度。
  5. 教训：流程上缺什么（Runbook、开关、监控缺口）。

文化：追责个人 不利于 真实信息上浮；追责 系统与流程 才能减少复发。

六、On-Call 健康：避免「永远同一个人」

  • 轮值表 与 时区 明确；交接班 简短书面记录。
  • 告警降噪（见监控篇）：减少 无行动告警。
  • 事后调休 或 补偿（视公司政策）：长期透支会流失骨干。

七、与系列其它文章的关系

  • 告警与 SLO：见《企业级流量监控与可观测性：指标、日志、追踪与 SLO》。
  • 回滚与发布：见《企业级发布策略：蓝绿、金丝雀、滚动与特性开关》。
  • 流水线与变更审计：见《企业级 CI/CD：流水线设计、门禁与发布治理》。

八、收束

企业级事故响应 = 分级清晰 + 角色明确 + Runbook 可执行 + 复盘可落地。技术栈会变，时间线与行动项闭环 不变。
