微前端实践：qiankun 生命周期、沙箱与数据通信

发表于2026-04-20|微前端
|阅读量:

qiankun 基于 single-spa，在蚂蚁场景打磨：HTML Entry、JS 沙箱、样式隔离、预加载。下面写 注册参数、子应用导出协议、沙箱类型与限制、全局状态，并带 配置示例。

一、主应用注册（概念）

**entry**：可以是 远程部署的 index.html，实现 独立部署。

二、子应用必须导出的生命周期

Webpack 子应用常用 package.json name 与 public-path 动态 __webpack_public_path__；Vite 需 插件 适配 UMD/system 导出（社区有 vite-plugin-qiankun 等）。

子入口 导出：

unmount 不干净 → 二次进入 重复监听、定时器泄漏。

三、沙箱：快照 vs Proxy

JS 沙箱 目标：子应用 改动 window.xxx 在 卸载后恢复，避免 污染下一子应用。

限制

  • 原生构造函数、document.body 直接 append 等 难完全回滚。
  • 第三方库 把东西挂 window 上 → 与 strict 沙箱 冲突时需 黑名单 或 换隔离策略。

样式隔离：scoped + experimentalStyleIsolation（类名前缀）或 Shadow DOM（注意 弹层）。

四、数据：props 与 initGlobalState

主应用 initGlobalState 可做 轻量共享；重业务 仍建议 BFF + 主应用鉴权。

反模式：把 整棵业务树 塞 global state → 类型与版本 难演进。

五、路由同步

子应用 router base 必须 与 activeRule 一致；history 模式 下 主应用 需 兜底 把 /vue/** forward 到子应用容器。

典型坑：子应用 自己改 window.location 导致 整页刷新 跳出微前端 → 统一用 router API。

六、收束

qiankun = single-spa + 工程化沙箱 + HTML Entry。落地成本在 子应用构建适配 与 样式/全局单例；先规范 unmount，再谈 prefetch。
