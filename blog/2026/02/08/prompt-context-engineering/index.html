AI工程实践

2026-02-08

很多人把「提示词」理解成几句咒语；在真实项目里，更可靠的说法是 上下文工程（Context Engineering）：你提供给模型的 文本、代码、工具返回、系统说明 的总和，决定了输出质量的上限——不是提示越长越好，而是信号越干净越好。

这篇分 上下文的构成、预算与噪声、可复制的写法、反模式、和 MCP/Skill 的组合，尽量写到能直接当检查表用。

一、上下文从哪来

  • 用户消息：需求、约束、追问、修正——唯一完全由你掌控的主通道。
  • 系统 / 开发者说明：角色、输出格式、安全边界（若客户端支持）。
  • 工作区与选中内容：打开的文件、diff、光标附近代码——模型「以为你正在看」的往往就这些。
  • 工具结果：终端输出、MCP 返回、检索到的文档片段——事实优先采信这里，模型总结是第二层。
  • 历史轮次：多轮对话里，越早的越可能被压缩或遗忘（依产品实现而定）。

铁律：模型不会自动知道你脑子里没写出来的前提；没写进上下文或没通过工具验证的，就当不存在。

二、上下文预算：质量 > 数量

常见误区是「我把整个仓库贴进去」。实际上：

  • 噪声会淹没信号：无关文件、过期注释、巨大生成的 lockfile 片段，都会拉高误判率。
  • 重复矛盾信息比「信息少」更糟：例如一处写用 Vue2，另一处贴 Vue3 示例，模型会「折中」出一个四不像。

实践：先写清 目标与边界，再按需 增量附加——附件、路径、接口名，而不是一次性倒垃圾。

三、几条硬规则（可直接当习惯）

  1. 先结论后细节：首段写 目标、验收标准、禁止项；背景放后面。
  2. 约束显式化：语言与运行时版本、框架、不能改的模块、兼容性底线（浏览器、Node 版本）。
  3. 给对比物：「当前行为 / 期望行为」「错误输出 / 正确输出」「慢的路径 vs 快的路径」。
  4. 控制范围：指定目录或文件列表；大重构拆成 多轮小 PR。
  5. 失败可复现：命令、环境变量（可打码）、最小代码；避免「就是不行」式描述。
  6. 指定输出形态：例如「先给步骤列表再给代码」「只改 src/foo 下文件」「用表格对比两种方案」。

四、结构化提示的轻量模板（不必死板）

可按任务类型选用：

Bug 修复

  • 现象（用户可见）
  • 复现步骤（最短）
  • 期望 vs 实际
  • 已尝试过的方向（避免重复绕圈）
  • 允许改的边界

功能开发

  • 用户故事或接口契约
  • 非目标（明确不做）
  • 数据模型与边界情况
  • 验收：测试用例、手工检查步骤

重构

  • 动机（性能、可读性、去重复）
  • 禁止行为（行为不变 / 可改公开 API 否）
  • 分阶段计划与每阶段验证命令

五、反模式（踩过坑的会更熟）

  • 含糊动词：「优化一下」「更优雅」——没有指标就没有完成定义。
  • 超载附件：十个无关文件 + 五千行日志——应先 裁剪到相关区间。
  • 隐含业务规则：权限、计费、合规若不写，模型会按「常见互联网产品」瞎补。
  • 迷信单次长回答：复杂任务应 多轮，每轮带 新的工具结果或测试结果 迭代。

六、和 MCP、Skill 的关系

  • MCP：把 可验证的事实与受控动作 接进来（查库、工单、内部文档站）。适合回答「现在系统里到底是什么」。
  • Skill：把 团队惯例与 SOP 固化成文档，减少每轮重复说明。

二者都不替代 你把目标与边界说清楚；它们负责让「说清楚」可以 复用、组合、版本化。

组合思路：重复出现的 前置交代 → 沉淀 Skill；重复要查的 数据源 → 接 MCP；对话本身只保留 本次任务特异 的部分——这样提示会越来越短，而 上下文更干净。

七、怎么判断「这次上下文工程做得好不好」

事后可以自问：

  • 模型是否 少问 了本可在第一条消息里写清的澄清？
  • 输出是否 可验证（命令、测试、diff 范围）？
  • 是否 引用或执行了 工具结果，而不是纯编？
  • 若重来，你会 删掉 哪些附件、补充 哪一句约束？

把每次高效对话里 重复的前置块 挪进 Skill，把 重复要查的数据 挪进 MCP，下一轮就会更轻松。更偏 Agent 多步与工具调用 的内容，见《检索增强与 Agent：工具调用在干什么》。
