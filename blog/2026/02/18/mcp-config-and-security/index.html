MCPAI工程实践

2026-02-18

上一篇《MCP 是什么》偏概念；这篇偏 动手：客户端里大概怎么配、stdio 与远程各适合什么、安全与排障 时盯哪些点。具体 JSON 字段名以你所用客户端与 MCP 版本为准，这里给的是 结构级 示例，便于你对照文档改。

一、你在配置里到底写了什么

对多数编辑器类 Host 而言，配置通常表达三件事：

  1. 如何启动 Server：命令、参数、工作目录。
  2. 环境：API Key、数据库 URL、是否只读等 通过 env 注入，避免写死在仓库（尤其不要提交密钥）。
  3. 传输方式：本地子进程（stdio）或 URL（HTTP/SSE）。

一个 虚构的、仅作形状参考 的片段（勿直接复制为生产配置）：

{
  "mcpServers": {
    "company-docs": {
      "command": "npx",
      "args": ["-y", "@example/mcp-docs-server", "--base", "https://docs.internal.example"],
      "env": {
        "DOCS_TOKEN": "${env:DOCS_TOKEN}"
      }
    }
  }
}

要点：

  • 命令能在终端独立跑通 再交给 Host；Host 只是把同样的事自动化。
  • 密钥用 环境变量占位 或系统密钥链，别写进 Git。

二、stdio：本地子进程模式

形态：Host 拉起一个进程，stdin/stdout 当信道；Server 通常是 Node、Python、Go 写的 CLI。

优点

  • 部署简单，适合个人脚本与小团队封装。
  • 数据可限制在本机进程边界内（仍要注意 Server 自己会不会把数据外传）。

缺点

  • 每个 Host 实例各自拉起进程，资源与版本要心里有数。
  • 排障要看 子进程日志（有的 Server 默认打 stderr）。

适用：读本地受限目录、包装内部 CLI、只读查询类工具。

三、HTTP / SSE：远程服务模式

形态：MCP Server 作为常驻服务，客户端用 HTTP（常配合 SSE 做流式）连接。

优点

  • 多客户端共享；便于 集中鉴权、限流、审计。
  • 与 API 网关、零信任 等企业设施对齐。

缺点

  • 运维成本高；网络与超时让 错误处理 更关键。

适用：全公司共用的「查工单 / 查人 / 查发布状态」类能力。

四、安全清单（建议打印贴在团队 Wiki）

  1. 最小权限：Server 里读哪些路径、调哪些 API、能否写——默认 只读，写操作单独开关。
  2. 密钥：短生命周期 token、轮换；日志里 脱敏。
  3. 租户隔离：多团队共用时，工具参数里要带 租户/环境，避免串数据。
  4. 副作用与幂等：创建、扣款、发消息类工具要有 幂等键 或确认步骤。
  5. 输出截断：大结果分页或摘要，防止撑爆模型上下文或泄露过多数据。
  6. 审计：谁在什么时间调用了什么（至少对写操作）；便于事后追责与复盘。

五、排障：常见「配了但不生效」

现象可能原因
Server 起不来本机无 Node/Python、路径错、npx 网络受限
工具列表为空握手失败、Server 未实现 list tools、版本不兼容
调用超时下游 API 慢、未设合理 timeout；应考虑异步或分页
模型「像调了工具」但事实不对只看总结没看 原始 tool result；或 Server 返回了错误但被模型弱化

习惯：重要结论以 工具返回原文 + 你自己的一次独立验证 为准。

六、和「自己写 HTTP 给模型调」的取舍

  • 快速业务脚本：直连内部 HTTP 往往最快，但要自己处理鉴权、截断、错误语义。
  • 多人、多工具、长期维护：MCP Server 把 契约 固化，换 Host 少改代码。

没有银弹；团队小、工具少时不必强行 MCP，但一旦工具超过两三个、且要在多个客户端复用，协议化的收益会明显起来。

七、小结

MCP 的价值在 标准契约 + 可审计的动作边界。配置阶段把 启动方式、环境、权限 想清，比纠结「提示词怎么写」更能避免线上事故。概念篇见《MCP 是什么》；和 Agent、RAG 的配合见《检索增强与 Agent：工具调用在干什么》。
