Git 约定式提交

name: git-conventional-commit
description（建议保持一行摘要给模型发现用）: 根据变更内容与 diff 撰写符合 Conventional Commits 的提交说明（type/scope/subject/body/footer）。在用户请求写 commit message、整理提交、或准备 push/PR 时使用。

原则

  • 一条提交只做一件事；若改动混杂，建议拆 commit 或提醒用户拆分。
  • subject 用简体中文或英文与仓库习惯一致，50 字左右，动词开头，不要句号结尾。
  • body 说明「为什么」和「要注意的影响」；footer 写 BREAKING CHANGE:、Fixes #123 等。

Type（常用）

type 含义
feat 新功能
fix 缺陷修复
docs 文档
style 格式（不影响逻辑）
refactor 重构（无功能变更）
perf 性能
test 测试
chore 构建/工具等非 src 变更
ci CI 配置

格式模板

示例：

输出方式

  • 若用户未指定语言，与对话语言或仓库 README 语言一致。
  • 给出 一条推荐 subject，必要时再给 备选；正文与 breaking 变更单独成段。

系统化排错

name: systematic-debugging
description: 用可复现步骤、假设验证与最小化定位系统性排查 bug（日志、二分、环境差异）。在用户描述报错、间歇性问题、无法复现、或要求「帮我查原因」时使用。

流程（按顺序）

  1. 确认现象：期望 vs 实际；错误信息完整粘贴；是否偶发、与数据/环境是否相关。
  2. 稳定复现：最小操作路径；若不能复现，记录时间、用户、请求 ID、版本、配置差异。
  3. 形成假设：列出 2–3 个最可能原因，每个都要能用实验证伪。
  4. 验证：加临时日志（入参/分支/返回值）、缩小调用范围、注释法隔离；必要时 git bisect。
  5. 修复与回归：修一处；补测试或复现脚本；检查同类代码路径。

约束

  • 优先 读栈与第一处抛错点，勿在无关层反复改。
  • 修改前 用工具/命令确认（读文件、跑测试、看配置），避免纯猜。
  • 间歇性问题查：超时、重试、竞态、缓存、时钟、资源耗尽。

输出结构

向用户说明时使用：

  • 根因（一句话）
  • 证据（日志、代码位置、复现步骤）
  • 修复（改了什么）
  • 如何防止再犯（测试、监控、文档）

PR 代码评审

name: pr-code-review
description: 按风险与可维护性评审 PR：正确性、安全、性能、测试、边界与可观测性。在用户发起 code review、看 diff、或评估合并风险时使用。

必读项

  • 正确性：边界条件、空值、并发、错误路径是否覆盖。
  • 安全：注入、权限、敏感数据日志、依赖解析不可信输入。
  • API/契约：破坏性变更是否标注；版本与文档是否一致。
  • 性能：N+1、不必要的全量加载、大对象序列化、热路径分配。
  • 可维护性：命名、重复逻辑、魔法数、过大函数/文件。

反馈分级

  • 阻塞：会导致错误数据、安全漏洞、线上明显故障。
  • 建议：可读性、结构、应在跟进中处理的问题。
  • 可选：风格 nit、与未来重构相关的想法。

回复格式

对每个问题：位置（文件/函数）→ 问题 → 建议或示例；优点可简短列 1–2 条。避免泛泛而谈。

若信息不足

主动说明需要：相关 issue、迁移步骤、压测结果、feature flag 策略等。

安全重构

name: safe-refactor
description: 指导小步重构：先保行为再改结构；测试、特征测试与回归策略。在用户要「重构、清理、抽函数、改名、搬文件」时使用。

顺序

  1. 绿：现有测试能通过；关键路径无测试时，先加 特征测试（记录输入输出或大段快照）。
  2. 小步：一次只做一个机械性变更（rename / extract / move），每步可提交。
  3. 行为不变：不顺便改业务规则；业务与结构拆开两次 PR 更清晰。
  4. 验证：跑相关测试与手动冒烟；涉及 API 时检查调用方。

适用手法

  • 抽函数：命名表达意图；参数最小化；副作用集中到少数处。
  • 搬移：先改引用再删旧实现；Git 下可用「重命名检测友好」的提交顺序。
  • 删除死代码：确认无动态引用（字符串反射、配置、构建裁剪）再删。

避免

  • 大面积格式化与逻辑变更同一 commit。
  • 无测试的大跨度重写；应用 绞杀者模式 或分模块替换。

REST/HTTP API 卫生

name: rest-api-hygiene
description: REST/HTTP API 的命名、状态码、幂等、分页、版本与错误体等实践清单。在设计或评审 API、OpenAPI、或前后端契约时使用。

资源与 URL

  • 名词复数：/users、/orders；用 嵌套表达从属：/users/{id}/orders。
  • 动词 用 HTTP 方法表达：GET 查询、POST 创建、PUT/PATCH 更新、DELETE 删除。
  • 避免 RPC 式路径泛滥（如 /getUser）；特殊动作可子资源：/orders/{id}/cancel。

状态码（常见约定）

场景 码
成功返回实体 200
成功创建 201 + Location
无 body 成功 204
客户端参数错 400
未认证 401
无权限 403
不存在 404
冲突（重复创建等） 409
服务端错 500

错误体

  • 统一结构：code（机读）、message（人读）、details（字段级可选）。
  • 不要 泄漏栈跟踪给客户端；日志里可追溯 requestId。

幂等与安全

  • POST：非幂等；必要时 Idempotency-Key。
  • PUT：整体替换幂等；PATCH 部分更新需文档字段语义。
  • 分页：cursor 或 page+size 二选一并文档化；返回 next_cursor / total 视需求。

版本

  • URL 前缀（/v1/...）或 header；破坏性变更升版本，旧版保留窗口期需说明。

SQL 安全

name: sql-security
description: 防范 SQL 注入与越权：参数绑定、动态标识符白名单、最小权限与敏感查询审计。在用户编写原生 SQL、拼字符串查询、评审 DAO/Repository、或讨论数据库安全时使用。

注入防护（必须）

  • 永远不要将用户输入直接拼进 SQL 字符串（含 ORDER BY、表名、列名、LIMIT 数值若来自不可信源）。
  • 值一律用 参数绑定（prepared statements / ORM 占位符）；不同语言驱动用法不同，原则是 结构与数据分离。
  • 动态表名/列名：仅用服务端 固定白名单 映射（allowlist[userChoice] → 合法标识符），禁止拼接 Raw 字符串。

权限与数据暴露

  • 应用连接 DB 的用户 最小权限（禁止生产应用账号 SUPER/FILE 等）；读分离账号只读。
  • 避免 SELECT * 经接口泄露宽表；敏感列加密或脱敏在应用层策略文档化。
  • 批量任务、迁移脚本 单独账号，与线上 OLTP 账号隔离。

业务侧 SQL 风险

  • IDOR：仅靠递增 id 查记录时要校验 资源归属（租户/用户 id）。
  • 多语句/存储过程：关闭不必要的多语句执行；调用 proc 仍要参数化。
  • 模糊查询：LIKE 用户前缀时注意索引与 通配符滥用 带来的 DoS（过长 pattern）。

评审清单

  • 无字符串拼接 SQL
  • 动态标识符来自白名单
  • 权限与租户隔离在查询条件或中间层 enforced
  • 错误信息对外不暴露库表细节（对内日志可详细）

Web 安全基础

name: web-security-basics
description: Web 常见漏洞与防护：XSS、CSRF、CSP、Cookie 属性、开放重定向、敏感接口鉴权。在用户写表单/登录、富文本、OAuth 回调、或做安全评审、渗透前自检时使用。

XSS（跨站脚本）

  • 输出编码：按上下文转义（HTML/JS/URL/CSS）；框架默认绑定优先。
  • 富文本用 白名单净化（成熟库），禁止仅靠黑名单。
  • CSP：Content-Security-Policy 限制脚本来源；nonce/hash 收紧内联脚本。
  • dangerouslySetInnerHTML / v-html：仅可信或已净化内容。

CSRF

  • 状态变更请求：CSRF Token、SameSite Cookie（Lax/Strict 按场景）、关键操作用 二次确认/Re-auth。
  • GET 不可有副作用；CORS 不替代 CSRF 防护（Cookie 场景）。

Cookie 与会话

  • HttpOnly、Secure、SameSite 默认开启；会话固定攻击：登录后轮换 session id。

其他常见坑

  • 开放重定向：next/redirect 参数 白名单域 或相对路径校验。
  • 点击劫持：敏感页 X-Frame-Options 或 CSP frame-ancestors。
  • 目录遍历、文件上传：校验扩展名与 MIME、存储隔离、唯一文件名；不执行上传目录脚本。

序列化与 SSRF

  • 反序列化不可信输入用安全格式；服务端请求 禁止 根据用户输入随意请求内网 URL（SSRF）。

可观测性基础

name: observability-basics
description: 结构化日志、指标、链路与 requestId/correlation id 的基础实践与噪声控制。在用户加入监控、排查线上、讨论「可观测性」、或评审日志/埋点时使用。

日志

  • 结构化（JSON/key-value）：level、timestamp、message、**requestId**、service、error.type、error.stack（仅内部）。
  • 上下文：同请求内复用 requestId；异步任务继承或生成 jobId。
  • 禁止：密码、token、完整卡号、证件号；必要字段 脱敏（尾号、hash）。
  • 级别：ERROR 需可操作；避免热路径 DEBUG 刷屏；采样需可配置。

指标（Metrics）

  • RED（面向请求）：Rate、Errors、Duration；关键依赖（DB、缓存、MQ）设 延迟与饱和。
  • 命名统一：{subsystem}_{action}_{unit} 或与团队约定一致；标签基数 勿爆炸（避免无界 userId 作 label）。

链路（Tracing）

  • 跨服务传递 trace context（W3C Trace Context）；Span 命名体现 业务操作 而非仅函数名。
  • 分布式事务/重试在 trace 上应 可辨认。

排错输出

向用户说明根因时引用：requestId、时间范围、关键日志字段、关联指标/图；敏感信息打码。

依赖与供应链

name: dependency-supply-chain
description: 依赖与供应链：锁文件、版本约束、审计命令、可疑包与最小依赖面；容器与 SBOM 入门。在用户安装 npm/pip/go mod、升级依赖、CI audit、或讨论投毒包、供应链攻击时使用。

基线

  • 锁文件提交：package-lock.json / pnpm-lock.yaml / poetry.lock / go.sum 等，保证可复现构建。
  • 版本范围：生产依赖优先 精确或窄范围；大版本升级单开 PR + 变更说明。

漏洞与恶意包

  • 定期运行 npm audit / pnpm audit、pip-audit、trivy 等；区分 可利用 与 噪音，按严重度排期。
  • 安装前看：下载量维护、仓库链接、近期发布是否异常；避免复制粘贴「一行 curl 装未知脚本」。

减少攻击面

  • 删未使用依赖；同能力优先 标准库/单体模块。
  • 可选依赖用 optional / feature flag；生产镜像 多阶段构建 仅含运行所需。

进阶

  • 关键项目记录 SBOM；CI 里对镜像与锁文件做 静态扫描；私有注册表与 完整性校验（sigstore/npm provenance 等）按团队成熟度启用。

限流与可靠性

name: rate-limiting-reliability
description: API 限流、重试、退避、幂等与防刷；保护下游与公平使用；客户端与服务端协同。在设计公开 API、网关、爬虫友好、429/Retry-After、或讨论「冲垮服务」时使用。

服务端

  • 限流维度：IP、用户 id、API Key、租户；登录/验证码等 更严。
  • 算法选型：固定窗口简单；漏桶/令牌桶平滑；分布式用 Redis 等并注意 原子性。
  • 响应：429 Too Many Requests，可选 **Retry-After**（秒）；文档说明配额与补全方式。
  • 超额成本：读接口可缓存；写接口 队列化 或 异步。

客户端与重试

  • 重试仅对 幂等 或 已做去重 的请求；退避 exponential backoff + jitter。
  • 尊重 429/503 + Retry-After；勿无限重试。

幂等与去重

  • 写操作支持 Idempotency-Key 或业务唯一键去重，避免重试导致 重复下单/扣款。

人机与滥用

  • 高价值接口：验证码、设备指纹、风控分层（与合规团队对齐）；日志审计异常模式。

缓存与一致性

name: caching-consistency
description: 缓存键设计、TTL、失效策略、惊群与一致性取舍；HTTP/CDN 与应用层缓存协作。在用户加 Redis/CDN、性能优化、或出现「读到旧数据」「缓存雪崩」时使用。

键与命名

  • 键包含版本或 schema：{prefix}:{id}:{version}；避免歧义碰撞。
  • 大规模列表：分页或游标 入键；注意键 基数 与内存上限。

失效

  • TTL：结合业务容忍延迟设；热点可 主动失效 + 短 TTL 双保险。
  • 更新路径：写库后 删缓存 或 写穿/写回 策略在全项目统一；避免「只写 DB 忘删缓存」。
  • 惊群：热点 key 过期时 单飞（mutex/互斥重建）或 永不过期 + 异步刷新。

一致性层级

  • 接受 最终一致 的场景才可大胆缓存；强一致 读路径避免多级缓存各论；金融类按领域规则设计。

HTTP/CDN

  • Cache-Control、ETag、Vary 正确设置；需登录的接口 默认 no-store；版本化静态资源文件名利于长缓存。

前端可访问性（a11y）

name: frontend-a11y
description: Web 可访问性：语义 HTML、键盘操作、焦点、对比度、表单与 ARIA 使用规范。在用户做页面/UI、组件库、或提到 WCAG、读屏、键盘导航时使用。

结构与语义

  • 控件用对的元素：需要点击用 <button> 而非 <div onclick>；导航用 <nav>，主内容 <main>。
  • 标题层级 h1–h6 有序；页面有唯一主标题思路。
  • 图片 提供有意义 alt；装饰图 alt=""。

键盘与焦点

  • 可交互元素 Tab 可达；可见焦点环（勿 outline: none 而不替代）。
  • 自定义下拉/对话框：焦点陷阱、Escape 关闭、aria-modal / role 配对文档查阅。
  • 跳过重复导航可提供 跳转链接（skip link）。

表单

  • label 关联 input（for/id 或包裹）；错误信息用 aria-describedby 或 live region 通知。
  • 必填在文案与 aria-required 上一致。

ARIA 原则

  • 少的 ARIA：优先修正语义 HTML；aria-* 补缺失语义，不重复原生属性。
  • 动态更新区域：aria-live 适度，避免刷屏。

视觉

  • 文本与背景 对比度 达标；勿仅靠颜色传达状态（配图标/文案）。

密钥与配置

name: secrets-and-config
description: 密钥与环境变量：存放位置、轮转、.env 与 CI、十二因子配置；避免泄漏到仓库与日志。在用户配置 API Key、数据库 URL、K8s Secret、或评审 Dockerfile/CI 时使用。

存放与注入

  • 密钥不进 Git：用环境变量、密钥管理服务（云 KMS/Parameter Store/Vault）或运行时挂载。
  • .env 仅本地，.env.example 给键名与假值说明；.gitignore 覆盖常见 secret 文件。
  • 分环境：dev/staging/prod 分离凭据；生产 禁止 复用开发密钥。

运行时

  • 进程内读取 缓存 即可，不要 print 完整连接串到日志。
  • 前端 bundle 无密钥：OAuth client_secret 仅后端；浏览器可见的仅为公开 client_id。
  • CI：用平台 encrypted secrets；fork PR 不泄漏上游 secret（按平台默认行为审查）。

轮转与应急

  • 支持 无停机密钥轮换（双凭证窗口或平滑 reload）。
  • 泄漏处置：吊销、轮换、审计访问日志；提交历史中的密钥需 视为永久泄漏 并轮换。

配置与代码分离

  • 十二因子：配置属环境，同一制品多环境部署；功能开关优先配置中心，硬编码默认值需文档化。

个人信息与最小化

name: pii-data-minimization
description: 个人数据最小化、脱敏、留存与跨境合规要点（日志、导出、备份、第三方共享）。在用户存用户资料、打日志、数据导出、GDPR/个保法相关讨论时使用。

原则

  • 只收集业务必需的字段；能推导的不重复存；定期审查宽表字段必要性。
  • 目的限定：采集时声明用途；二次使用需法律依据或授权。

存储与展示

  • 敏感字段 加密或哈希（可检索场景用确定性加密或 tokenization）；展示 脱敏（手机/证件中间位）。
  • 权限：按角色与场景裁剪字段（API DTO 分层）；管理后台操作 审计日志。

日志与观测

  • 默认不打 PII；必须打时用 假名化 id 或脱敏；聚合分析避免原始明细落盘。

留存与删除

  • 定义 保留期限 与删除流程（含备份、冷库、第三方处理器）；用户 删除/导出 请求有 SLA。
  • 跨境：数据出境与安全评估按法务结论执行，技术层标注数据驻留区域。

第三方

  • 与供应商共享数据：DPA/合同、子处理者名单、泄露通报条款；最小权限 API。
