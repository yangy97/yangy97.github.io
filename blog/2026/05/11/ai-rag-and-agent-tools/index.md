这篇把三条常一起出现但容易混的概念拆开：==RAG（检索增强生成）==、**Agent（多步自主或半自主循环）**、**工具调用（Tool Use）**。它们可以单独用，也可以组合；组合方式决定了 **自动化程度、风险与成本**。

***

### 一、RAG：先检索，再生成

\==问题==：模型权重里的知识有截止日期，且对 **你们私有文档、最新接口、内部规范**  coverage 不足；长上下文全塞进去又贵又噪。

**思路**：把知识切成 **片段（chunk）**，用向量或关键词检索出 Top-K，再把这些片段作为 **证据** 放进提示，让模型基于证据作答。

**典型流水线（简化）**

1. **切分**：按标题、段落、代码块边界切 chunk；太小丢语义，太大检索不精。
2. **嵌入（Embedding）**：把 chunk 变成向量，存进向量库。
3. **检索**：用户问题也向量化，做相似度搜索；可加 **关键词/BM25** 混合。
4. **重排序（Rerank）**：用小模型对候选片段再排一次，提高 precision。
5. **生成**：把「问题 + 检索片段 + 引用要求」交给 LLM；输出里要求 **标明引用** 便于核对。

**常见坑**

* 片段里 **缺主语**（「如上所述」），检索到了也读不懂——切分与元数据（文件名、章节路径）很重要。
* **重复与矛盾文档** 会误导模型——需要版本戳或「以某页为准」的规则。
* RAG **不是**「绝对不胡编」，而是 **降低胡编概率**；关键结论仍要人或用工具二次确认。

***

### 二、Agent：多步「感知—规划—行动—观察」

**问题**：复杂任务无法一次写完；需要 **读文件、跑命令、看结果、再决定下一步**。

**典型循环**

1. **感知**：当前目标、已有上下文、工具返回。
2. **规划**：下一步调用哪个工具或改哪个文件（可由模型或规则引擎产生）。
3. **行动**：执行工具或应用 patch。
4. **观察**：把 stdout、diff、HTTP 状态码 **回灌** 进上下文。
5. 直到满足终止条件（测试通过、用户确认、步数上限）。

**与「单次补全」的差别**：Agent 把 **反馈闭环** 打开了；代价是 **更长的链路、更高的失控面**——需要 **步数上限、权限、人在环（human-in-the-loop）**。

***

### 三、工具调用：把「能力」变成可调函数

无论叫 function calling 还是 tool use，本质都是：**模型输出结构化指令** → **运行时执行** → **结果写回上下文**。

**和 MCP 的关系**：MCP 是 **多种工具的一种标准化交付方式**；没有 MCP 时你也可以手写 HTTP Tool，只是互操作性差一些。

**设计工具时的原则**

* **名字与参数自解释**；避免「万能 tool」一个字符串塞所有。
* **只读与写入分离**；写入类默认要确认或幂等。
* **返回结构稳定**：便于模型解析，也便于日志与测试。

***

### 四、三者怎么组合（常见配方）

| 配方 | 做法 | 适用 |
|------|------|------|
| **RAG only** | 文档问答、规范查询 | 低风险、以读为主 |
| **Tool only** | 查实时数据、跑测试 | 要准确状态，不必向量库 |
| **Agent + Tools** | 改代码、多文件重构、排障 | 中高风险，要约束与验收 |
| **Agent + RAG + Tools** | 大仓库里「先找文档再改代码再跑测」 | 能力最强，工程复杂度也最高 |

***

### 五、落地时的工程要点

* **可观测性**：每步工具名、参数、返回摘要、耗时——否则出事故只能猜。
* **预算**：Agent 多轮 × 长上下文 = **费用与延迟**；要对齐业务预期。
* **人在环**：发布、删数据、对外发信等，**默认要人点确认**。
* **评估**：准备固定任务集（golden tasks），改版模型或提示时 **回归对比**。
