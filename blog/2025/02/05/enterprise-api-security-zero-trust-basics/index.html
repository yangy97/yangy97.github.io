企业级 API 安全与流量治理：鉴权、限流与常见漏洞面

约 1015 字大约 3 分钟

安全API网关企业级

2025-02-05

对外 API、对内微服务 HTTP，在企业里通常要同时满足：身份可信、权限最小、滥用可挡、出事可追溯。下面按 鉴权模式、网关侧限流、幂等与重放、常见漏洞面 展开，并附 可落地的示例与检查清单。

一、鉴权放哪一层：网关 vs 应用

做法优点注意
网关校验 JWT / mTLS统一策略、早拒绝声明与业务角色仍需应用二次校验
仅应用校验灵活每个服务重复实现，易不一致
API Key（B2B）简单密钥轮换、按租户配额、审计

常见组合：公网 OAuth2/OIDC + JWT；内网服务间 mTLS + 服务身份（SPIFFE 等）；开放平台 API Key + 签名。

二、示例：JWT 在网关上「验签」之后，应用还要做什么

网关验证 签名、iss、aud、exp 后，把 sub / roles 以 Header 传给上游。应用层 仍必须：

  • 校验 资源级权限（例如「只能删自己的订单」不能仅靠 role=user）。
  • 不信任客户端传的 用户 ID Body，以 token 中的 sub 为准。

反例：仅网关验 JWT，应用接口 DELETE /orders/:id 不查 id 是否属于当前用户 → IDOR（不安全直接对象引用）。

三、限流：按什么维度、返回什么

企业里常见维度：IP、用户 ID、租户 ID、API Key、路由。
响应建议：429 Too Many Requests，Header 带 Retry-After 或 X-RateLimit-*（与客户端约定）。

示例：伪配置（思想）

# 示意：某 API 网关 / Envoy ratelimit 风格
routes:
  - match: { prefix: "/v1/orders" }
    ratelimit:
      per_tenant: 300/min
      burst: 50
  - match: { prefix: "/v1/login" }
    ratelimit:
      per_ip: 10/min   # 防撞库

要点：登录、验证码、找回密码 等路径要 更严；全局限流与 按租户 限流叠加，避免 大租户拖垮共享集群。

四、幂等、重放与「业务安全」

  • 写接口 应支持 幂等键（Idempotency-Key），网关或应用层去重，避免 重试风暴导致重复下单。
  • 敏感操作（转账、改权限）除 JWT 外可叠加 二次确认、风控评分、设备指纹（视行业）。

示例：客户端网络抖动对 POST /payments 重试三次，若无幂等键，可能 三笔扣款。服务端以 (user_id, idempotency_key) 唯一索引 拒绝重复执行。

五、常见漏洞面（与 API 相关）

  1. 越权：水平（他人资源）、垂直（普通用户调管理接口）。
  2. 批量赋值：PUT /user 体里带 role: admin 被全量更新。
  3. SSRF：服务端根据用户 URL 去拉取内网资源。
  4. 注入：SQL、NoSQL、命令注入（即便用 ORM 也要警惕原生拼接）。
  5. 敏感信息泄露：错误栈、日志里的 token、过度详细的 枚举用户存在 的提示。

治理：SAST/DAST、依赖漏洞扫描（见 CI/CD 篇）、定期渗透测试；设计评审时带 威胁建模 简版（STRIDE）。

六、零信任在 API 侧的朴素理解

不是「内网就可信」，而是：每次请求都带证据（身份、设备、上下文）、最小授权、全程加密与审计。

示例：内网服务从「VPC 互通即调用」改为 mTLS + 每服务身份，配合 网络策略 缩小横向移动面。成本是 证书与 Sidecar/网关运维；中小团队可先在 南北向（入口） 做强鉴权与 TLS，东西向 逐步演进。

八、收束

API 安全 = 网关统一策略 + 应用资源级授权 + 限流与幂等 + 持续扫描与复盘。先把 鉴权边界和 IDOR 讲清楚，再叠 零信任 与 细粒度配额。
