企业级发布策略：蓝绿、金丝雀、滚动与特性开关

约 678 字大约 2 分钟

发布Kubernetes企业级

2025-09-14

上线方式决定 故障爆炸半径。滚动（Rolling） 最省事；蓝绿 换流量快、回滚快；金丝雀 用小流量验证新版本；特性开关（Feature Flag） 把「发代码」与「对用户可见」拆开。下面写 适用场景、与入口层配合、数据与回滚注意点。

一、四种方式对比

策略资源占用回滚速度风险暴露典型依赖
Rolling低中（等新 Pod 就绪）中（新旧混跑一段时间）K8s 默认
Blue/Green高（双套）快（切流量）低（瞬时切换）另一套环境 + LB
Canary中快（停权重）低（流量小）Ingress 权重 / Mesh
Feature Flag低（一套代码）极快（关开关）依实现Flag 服务、SDK

二、滚动发布：默认选项要注意什么

  • maxUnavailable / maxSurge：控制 同时不可用 与 额外 Pod；高峰时要保守。
  • 就绪探针（readiness）：新 Pod 未就绪不接流量，避免 半启动 实例吃请求。
  • 优雅下线：preStop + 连接排空（Nginx draining），避免 切断长请求。

三、蓝绿：何时用

  • 需要 秒级回滚（金融、支付核心）。
  • 数据库 schema 已兼容 或 蓝绿各自独立库（成本高）。

切流量在 负载均衡 / DNS 或 Ingress 目标 Service 上完成；要保证 会话 不因切换而大面积登出（除非接受）。

四、金丝雀：与指标联动

流程示例：

  1. 新版本 5% 流量（Ingress 权重或 Mesh）。
  2. 观察 5xx、P99、业务错误率（对比基线）。
  3. 达标 → 25% → 50% → 100%；异常 → 回滚权重为 0。

关键：指标来自 网关 + 应用 + 业务；仅看 CPU 不够。

五、特性开关：与部署解耦

  • 发布：代码已在 prod，但 默认关闭。
  • 放量：按用户 ID 百分比、地域、内部员工先行。
  • 紧急止血：关开关 比 整包回滚 快。

代价：开关分支要 清理，否则代码成 意大利面；需 治理（过期开关扫描）。

六、数据库与发布顺序（再强调）

先兼容迁移 → 发应用 → 再删字段（expand/contract）。
金丝雀阶段若新版本写 新列、老版本仍读旧逻辑，要确认 双写/兼容读 策略。

七、与 CI/CD 的衔接

  • Pipeline 部署 canary release → 等待自动化检查 → promote 或 rollback。
  • 工具示例方向：Argo Rollouts、Flagger（与 Prometheus 联动），具体以选型为准。

八、收束

选策略看 RTO/RPO 要求、资源预算、团队成熟度。小步金丝雀 + 强监控 往往是企业里 性价比最高 的默认路径；蓝绿留给 强合规或强回滚 场景。
