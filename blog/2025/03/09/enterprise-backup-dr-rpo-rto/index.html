企业级备份与容灾入门：RPO、RTO 与演练

约 902 字大约 3 分钟

容灾备份企业级

2025-03-09

备份不是「每天跑一次脚本心里就踏实」，容灾也不是「多买两个区」。企业里通常要先谈清 RPO（能丢多少数据）、RTO（能停多久业务）、成本与复杂度，再选 备份频率、复制架构、切换流程。下面用 概念 + 简例 + 演练要点 铺一层入门深度。

一、RPO / RTO 是什么

指标含义例子
RPO（Recovery Point Objective）灾难发生后，允许丢失的数据时间窗口RPO=1h → 最多接受恢复到 1 小时前的快照
RTO（Recovery Time Objective）从灾难声明到 服务恢复可用 的时间上限RTO=4h → 4 小时内要切完流量、校验数据

直觉：RPO 越严 → 备份/复制越频繁或同步；RTO 越严 → 自动化切换、热备、多活 成本越高。

二、示例：不同业务线的「合理」目标（虚构）

系统典型 RPO典型 RTO常见手段
订单与支付核心分钟级～近零分钟～小时同步复制、多 AZ、严格演练
用户行为分析小时～天天异步备份、可重建
配置与特性开关近零分钟GitOps + 多副本存储

对齐方式：与 法务、财务、客服 一起定义「停服 X 小时 / 丢 Y 分钟数据」的业务损失，再反推技术方案，避免 工程师单方面拍脑袋。

三、备份：数据库与应用分别想什么

数据库

  • 全量 + 增量 / WAL 归档：恢复时能 点到时间点（PITR）。
  • 备份加密（静态与传输）、异地 存放、恢复测试（见下文）。
  • 云 RDS： often 一键快照，但要确认 跨区域复制 是否打开、保留期。

应用与配置

  • 无状态：镜像在仓库，配置在 Git（多环境篇），重建较快。
  • 有状态（上传文件、会话）：对象存储 版本控制 / 跨区域复制，或明确「丢失可接受」。

四、多 AZ vs 多 Region vs 多活

  • 同 Region 多 AZ：抵御 机房级 故障，通常 同步复制，RPO 接近 0（仍要读 SLA 细则）。
  • 跨 Region：抵御 区域性灾难，多为 异步复制，RPO 不为零。
  • 多活：双写或冲突解决复杂度高，仅核心业务 值得投入。

示例：某 SaaS 在 华东 主、华北 备；数据库 异步复制。若华东全挂，切换到华北时可能 丢失最近 N 分钟 写入——产品侧要有 对账与补单 流程，而不是假装「零丢失」。

五、演练：没有演练的预案≈没有预案

建议每年至少 桌面推演 + 一次真实恢复演练（可在隔离环境）：

  1. 随机抽 一份备份，按 Runbook 恢复到新实例。
  2. 校验 行数、校验和、抽样业务查询。
  3. 记录 实际耗时，与 RTO 对比，更新文档。

反例：备份任务显示「成功」，但 恢复时发现备份文件损坏——只在真演练时暴露。

六、与发布、监控的交叉

  • 误删数据、坏迁移：备份是最后防线；发布策略与 expand/contract 是第一道（见 CI/CD、发布策略篇）。
  • 切换瞬间：监控 双写延迟、复制滞后；告警 lag 超阈值要 阻断切流量或降级。

七、收束

企业级备份与容灾 = RPO/RTO 与业务对齐 + 备份可恢复 + 跨区策略清醒 + 定期演练。先从 「能否在文档规定时间内恢复一份库到测试环境」 做起，再谈 多活。
