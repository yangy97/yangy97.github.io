企业级 CI/CD 关心的不只是「能自动部署」，而是：==可重复、可审计、可回滚、可分环境验证==，并在 **质量 / 安全 / 合规** 上设 **门禁**。下面按 **流水线分层、制品与密钥、质量闸、发布策略、常见坑** 写深一点；工具名以 **GitLab CI / GitHub Actions / Jenkins** 为常见代表，语法需对照你们版本。

***

### 一、流水线典型阶段（逻辑分层）

| 阶段 | 目的 | 常见步骤 |
|------|------|----------|
| ==Source== | 可追溯 | 分支策略、Commit 关联工单、签名提交（可选） |
| **Build** | 可复现制品 | 安装依赖、编译、单测、**确定性构建**（锁文件） |
| **Static Analysis** | 早发现问题 | Lint、类型检查、SAST、Secret 扫描、依赖漏洞（SCA） |
| **Test** | 分层验证 | 单测 → 集成/契约 → E2E（可选 nightly） |
| **Package** | 不可变制品 | 镜像 tag（digest）、Helm chart 版本、SBOM |
| **Deploy（按环境）** | 可控上线 | dev → staging → prod；人工审批或策略审批 |
| **Post-deploy** | 闭环 | 冒烟、**合成监控**、指标对比、自动回滚（可选） |

**原则**：**越靠左越便宜**——在 PR 阶段拦住问题，比在生产救火便宜一个数量级。

***

### 二、制品（Artifact）与不可变部署

* **同一次构建**产出的镜像 / 静态资源 / Helm 包，**晋升（promote）** 到各环境，而不是每环境重新 build（避免「测的是 A、上的是 B」）。
* 镜像 tag 建议：**内容寻址**（digest）或 **`版本+构建号`**，避免只用 `latest`。
* **SBOM**（软件物料清单）：安全与许可证审计趋势，可与 **SCA** 工具集成。

***

### 三、密钥与配置：绝不写进仓库

| 做法 | 说明 |
|------|------|
| **CI 变量** | 敏感值用平台「密文变量」，按 **环境** 作用域划分 |
| **短期凭证** | OIDC 联邦登录云厂商（**无长期 AK/SK** 在 Runner） |
| **运行时注入** | K8s Secret / 外部密钥系统（Vault、云 KMS） |
| **配置分层** | `values-dev.yaml` / `values-prod.yaml`，**非敏感** 与 **敏感** 分离 |

**反模式**：把生产数据库 URL 明文写在 `docker-compose.yml` 并提交。

***

### 四、质量门禁（Quality Gates）示例思路

1. **测试**：覆盖率 **阈值**（按模块逐步提高）、关键路径 **必测**。
2. **安全**：高危漏洞 **阻断**、中危 **限期修复**。
3. **性能**：前端 **size-limit**（见性能系列）、API **性能基线**（可选）。
4. **合规**：许可证 **黑名单**、禁止依赖 **copyleft 误用**（视公司法务）。

门禁过严会导致 **绕过文化**——阈值要与团队共识，并 **定期复盘误杀率**。

***

### 五、数据库变更与 CI/CD

* **迁移脚本**（Flyway / Liquibase / 自研）与 **应用发布** 的先后顺序要有规范：常见 **先向后兼容迁移 → 发应用 → 再删字段**（expand/contract）。
* **大表变更**走 **在线 DDL** 策略或 **业务低峰窗口**，不要指望「一条 pipeline 无脑跑」。

***

### 六、发布策略（与接入层配合）

| 策略 | 含义 | 适用 |
|------|------|------|
| **Rolling** | 逐实例替换 | 默认、简单 |
| **Blue/Green** | 两套环境切换流量 | 快速回滚 |
| **Canary** | 小比例流量新版本 | 风险敏感、有指标支撑 |
| **Feature Flag** | 按用户/百分比开功能 | 与部署解耦，降低发版频率压力 |

金丝雀需要 **入口层权重** + **监控阈值自动晋升/回滚**（见《应用接入与流量转发》篇）。

***

### 七、Runner 与隔离

* **共享 Runner**：成本低，注意 **租户隔离**（容器、网络策略）。
* **专属 Runner**：合规或重型构建（Android/iOS、大前端 monorepo）。
* **缓存**：依赖缓存加速构建，但要 **缓存键** 与 **锁文件** 一致，避免脏缓存。

***

### 八、GitHub Actions 极简骨架（示意）

```yaml
name: ci
on: [push, pull_request]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - run: npm test
      - run: npm run build
```

生产部署 job 往往单独 **environment**，加 **required reviewers**。

### 九、与「企业级」其它能力的关系

* **流量怎么进服务**：网关、Ingress、负载均衡 → 见《企业级应用接入与流量转发》。
* **上线后怎么知道健康**：指标、日志、追踪 → 见《企业级流量监控与可观测性实践》。

### 十、收束

企业级 CI/CD = **流水线结构化 + 制品与密钥治理 + 门禁可执行 + 发布策略与数据迁移有章可循**。工具换皮不换骨：**先把流程和责任人写清**，再选 GitLab 还是 Argo CD。
