Node 主线程跑 JS；==要利用多核==、**隔离崩溃**、**跑别的语言二进制**，会用到 **child\_process**、**worker\_threads**、**cluster**。选型错会导致 **共享状态错乱**、**内存翻倍**、或 **IPC 瓶颈**。本篇给 **决策树** 与 **代价**。

***

### 一、先分清：你要解决什么

| 目标 | 倾向 |
|------|------|
| ==CPU 密集==（加解密、压缩、图像） | **Worker Threads** 或 **子进程池** |
| **跑 shell / 独立程序** | **spawn** |
| **多进程监听同一端口（榨干多核）** | **Cluster** 或 **进程管理器 + 多实例** |
| **崩溃隔离、语言混合** | **子进程** |

**不要**用 **大量子进程** 做简单并行 I/O——多数情况 **主线程 + 异步 I/O** 已够；进程有 **创建成本与内存开销**。

***

### 二、child\_process

* **`spawn`**：流式、适合 **大输出**、**长时间运行**。
* **`exec` / `execFile`**：缓冲整个输出，**有大小限制**，适合 **短命令**。
* **`fork`**：多一个 **Node 子进程 + IPC channel**，适合 **把任务挪出去**。

**注意**：**shell 注入**（拼接用户输入进命令行）；**stdio** 背压；**退出码**与 **signal** 处理。

***

### 三、Worker Threads

* **共享内存**：`SharedArrayBuffer` + **Atomics**（场景少但强）。
* **多数场景**：通过 **MessagePort** 传 **结构化克隆** 的数据（大对象有 **拷贝成本**）。
* **适合**：同一份代码里 **并行算**，又不想 **起完整子进程**。

**限制**：**不能**直接共享普通 JS 对象跨线程；**调试**比单进程难。

***

### 四、Cluster

* **多个 Node 进程** 各有一个 **事件循环**，**监听同一端口**（内核分发连接）。
* **Session 粘滞** 若依赖内存，要 **sticky session** 或 **外置存储（Redis）**。
* 现代部署里常用 **容器多副本 + K8s Service** 代替手写 Cluster，但 **原理相同**。

***

### 五、对比小结

| 方案 | 内存 | 通信 | 典型代价 |
|------|------|------|----------|
| **Worker** | 较低（同进程池） | MessagePort | 序列化、调试 |
| **子进程** | 高（每进程一份 V8） | IPC / pipe | 启动慢、协议设计 |
| **Cluster** | 高 | 无共享业务状态 | 多份业务代码、状态外置 |

***

### 六、小结

**I/O 多** → 先 **异步 + 限流 + Stream**。**CPU 多** → **Worker 或进程池**。**要端口级多核** → **Cluster 或多实例 + LB**。先 **量瓶颈** 再选模型，避免 **过早进程化**。
