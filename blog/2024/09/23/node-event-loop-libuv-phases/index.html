Node.jslibuv事件循环全栈

2024-09-23

Node 的「单线程」指的是 执行 JavaScript 的主线程；文件、网络、DNS 等 I/O 在 libuv 线程池 / 系统调用 里完成，完成后再把回调塞回 事件循环。搞清 阶段（phases） 和 微任务插队规则，才能解释 setTimeout、setImmediate、process.nextTick、Promise 谁先谁后，以及 为什么重 CPU 会拖死整个进程。

一、主线程到底在干什么

  1. 执行当前同步代码（一整段 script、函数调用栈）。
  2. 阶段推进：每个阶段取出该阶段队列里的回调执行。
  3. 阶段之间：会处理 微任务（见下）。
  4. 循环，直到没有可调度工作且没有待处理 I/O（简化说法；实际还有 ref/unref 等细节）。

误区：「Node 单线程所以慢」——慢的是把 CPU 算力占满的那段 JS，与 I/O 并发能力不是一回事。

二、libuv 各阶段（概念版，以官方文档为准）

常见阶段包括（名称与顺序记 直觉 即可，细节随版本可能调整）：

阶段典型内容
TimerssetTimeout / setInterval 到期的回调
Pending上一轮 setImmediate 等遗留（部分场景）
Poll拉取新的 I/O 事件；与定时器、立即执行等交互最密
ChecksetImmediate 的回调
Close如 socket.on('close', ...)

记忆点：setImmediate 设计为在 I/O 之后 的 check 阶段跑；和 setTimeout(fn, 0) 的相对顺序 在不在 I/O 回调里会不同——这是面试与排障常考点。

三、微任务：process.nextTick vs Promise

  • process.nextTick 队列 优先级 高于 Promise.then 的微任务队列；且 在阶段切换前 可能把 nextTick 跑光（若递归注册会 饿死 I/O）。
  • Promise / queueMicrotask 属于 微任务，通常在 当前阶段宏任务结束后、下一阶段前 清空一轮（具体与 HTML 规范在 Node 中的实现细节可查当前版本文档）。

实践：不要在 nextTick 里无限递归；业务逻辑优先用 Promise/async-await，nextTick 留给 框架级 的「让出当前阶段」需求。

四、CPU 密集 vs I/O 密集

类型表现对策
I/O 密集等磁盘/网络，CPU 空保持异步 API，注意 并发上限 与 背压
CPU 密集大循环、加解密、图像处理Worker Threads、子进程、或 独立服务；不要指望 拆成很多 setTimeout 就能并行算

V8 执行 JS 时 长时间不进入事件循环，定时器、心跳、TCP 保活都会 推迟。

五、setImmediate 与 setTimeout(0) 谁先

  • 不在 I/O 回调里：顺序 不保证（依赖系统定时器精度与当前负载）。
  • 在 I/O 回调里：通常 setImmediate 先于 setTimeout(0)（与 poll → check 的设计一致）。

线上若依赖「谁先谁后」，应 改写逻辑（显式串行、或单一调度源），不要依赖 未文档化的顺序。

六、小结

事件循环 = 宏任务阶段 + 微任务插队 + libuv 异步结果回流。排障时：先区分同步长任务 vs 真异步；再查 定时器/I/O/Immediate 是否被饿死；CPU 重活 移出主线程。
